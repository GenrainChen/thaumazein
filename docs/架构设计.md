# Thaumazein 架构设计

## 系统总览

Thaumazein 采用从抽象到物理的**三层架构**。顶层解读高级语言乃至自然语言，中间层以底层语言抽象物理设备，底层则是感知与操作的物理实体。信息自顶向下传递为指令，自底向上传递为数据。

```
┌───────────────────────────────────────────────────┐
│              高层 High Level                        │
│  高级编程语言 / 自然语言解读 · AI 推理 · 高级功能    │
├───────────────────────────────────────────────────┤
│              中间层 Middle Level                    │
│  汇编/C 语言 · 传感器/操作器型号抽象 · 驱动与通信    │
├───────────────────────────────────────────────────┤
│              底层 Physical Level                    │
│  物理实体 · 感知器 · 操作器 · 感知-操作一体设备      │
└───────────────────────────────────────────────────┘
```

### 智能分布原则："下载式智能"

系统的核心架构原则是**所有智能由 Hermes 创造，Bastion 只执行 Hermes 的产物，Terminal 只转换物理信号**。这类似于 PLC（可编程逻辑控制器）的模型：工程师在工作站设计控制逻辑，PLC 在现场忠实执行。

| 维度 | Hermes（设计师） | Bastion（包工头） | Terminal（工人） |
|---|---|---|---|
| 设计控制律？ | 是 | 否 | 否 |
| 设计推理模型？ | 是 | 否 | 否 |
| 设计安全规则？ | 是 | 否 | 否 |
| 执行控制律？ | 否 | 是（下载的） | 否 |
| 执行推理模型？ | 否 | 是（下载的） | 否 |
| 执行安全规则？ | 否 | 是（下载的） | 否 |
| 转换物理信号？ | 否 | 否 | 是 |

**行为包（Behavior Pack）** 是这一原则的实现机制：Hermes 将验证过的控制程序、推理模型、安全规则打包为自包含的部署单元，下发给 Bastion 本地执行。

### 层间关系

- **高层 → 中间层**：将自然语言或高级编程语言的意图分解为可执行的操作序列，逐级下发
- **中间层 → 底层**：将操作序列翻译为特定型号传感器/操作器的寄存器级指令
- **底层 → 中间层**：物理设备采集的原始信号上报，经驱动解析为结构化数据
- **中间层 → 高层**：结构化数据汇聚，上报给 Hermes 支撑决策与知识积累

### 三级控制时序

实时控制回路无法容忍 Hermes-Bastion 往返延迟，因此通过"本地执行 Hermes 程序"解决：

| 层级 | 频率 | 决策位置 | 示例 |
|---|---|---|---|
| 硬实时 | 100Hz-10kHz | Bastion（执行 Hermes 程序） | 电机 PID 控制、传感器采样 |
| 软实时 | 1Hz-100Hz | Bastion（执行 Hermes 策略） | 路径规划、传感器融合 |
| 非实时 | <1Hz | Hermes（在线智能） | 任务规划、系统重构、模型重训 |

## 产品线

三层架构映射为三类产品，各自承担不同层次的抽象与物理职责。

```
product/
├── hermes/      Hermes — 高层（所有智能的创造者）
├── bastion/     Bastion — 中间层 + 高层（智能产物的执行者）
└── terminal/    Terminal — 底层（物理信号的转换者）
```

### Hermes 赫尔墨斯 — 高层

> Ἑρμῆς — 希腊信使神，穿越神界与人界的唯一使者。

Hermes 是 Thaumazein 中**唯一面向人类**的产品，也是**唯一具备智能创造能力**的节点。它不触碰物理世界，但通过四大能力——思维导图、流程图、编程、仿真——对物理世界拥有元模型层面的主宰权。

四大能力是 Hermes 的核心设计工具。它们并非线性流水线，而是根据问题性质灵活组合的四个视角：

```
思维导图          流程图           编程            仿真
"有什么"         "怎么做"        "写成规则"      "跑起来看"
质料因剖分        动力因剖分       形式因实现       四因综合验证
```

四因关系说明：
- **思维导图**剖分质料因——系统由什么组成，空间结构是什么
- **流程图**剖分动力因——系统如何变化，时间序列是什么
- **编程**将形式因（概念抽象）和动力因（运行逻辑）实现为可执行的形式化规则
- **仿真**对所有四因的综合验证——在数学环境中检验形式、质料、动力是否符合目的

这四种工具相互补充，可双向追溯。取决于问题性质：有时问题核心是"有什么"（基于数据），思维导图主导；有时问题核心是"怎么干"（基于流程），流程图主导。

#### 思维导图：质料因可视化剖分

回答"系统由什么组成？"——对质料因进行可视化剖分，将系统的物质构成拆解为可审视的空间结构。在 Hermes 中用于梳理系统组成要素：Bastion 由哪些模块构成、Terminal 有哪些型号、传感器涉及哪些物理量。当问题核心是梳理构成时（基于数据），思维导图主导，流程图服务于它。

#### 流程图：动力因可视化剖分

回答"系统如何变化？"——对动力因进行可视化剖分，将系统的变化过程拆解为可追溯的时间序列。在 Hermes 中用于描述系统行为：控制指令如何经 Bastion 下发到 Terminal、传感器异常如何触发推理与响应。当问题核心是执行过程时（基于流程），流程图主导，思维导图服务于它。

#### 编程：实现硬件逻辑

编程是 Hermes 将人类意图注入系统的手段。通过代码定义硬件行为：传感器的采样策略、操作器的控制律、通信协议的握手规则、Bastion 的执行程序。这些代码经过仿真验证后，打包为行为包下发到 Bastion 执行。Terminal 不需要知道为什么采样，Bastion 不需要知道为什么执行——这些"为什么"存在于 Hermes 的代码中。

#### 仿真：对物理实体的抽象

仿真定义了硬件的"存在本身"——至少在数学意义上。通过建立物理系统的数学模型，Hermes 在纯计算环境中复现整个三层系统的动态行为。**仿真是物理世界的数学替身**：

- **验证**设计方案在部署前是否可行，避免物理试错成本
- **探索**极端工况下系统的行为，不必将设备置于危险中
- **优化**控制参数和策略，在数学空间中穷举物理世界中无法重复的实验

仿真也是行为包下发前的必经环节：Hermes 在仿真中验证行为包的正确性和安全性，确认无误后才部署到 Bastion。

#### 元模型和元数据

编程与仿真的结合产生**元模型**——物理系统的数学镜像，以及**元数据**——描述系统结构与行为的结构化描述信息。Hermes 不直接操控物理设备，而是：

1. 在仿真中构建并验证物理系统的数学模型（元模型）
2. 通过编程将验证过的控制逻辑形式化，并提取结构化的元数据
3. 将元模型与元数据打包为行为包，下发到 Bastion 由程序执行引擎加载运行

Hermes 控制的不是物理设备，而是**物理设备的控制规则**——控制的控制，抽象的抽象。它操作的是物理实体的数学镜像，而非物理实体本身。元模型定义系统的动态行为，元数据描述系统的静态结构，二者共同构成行为包的核心内容：一个自包含的、版本化的、可验证的部署单元，包含控制律、推理模型、安全规则、降级程序等 Hermes 创造的智能产物。

### Bastion 堡垒 — 中间层 + 高层

Bastion 是 Thaumazein 的**脊梁**，担当"包工头"的角色：它不设计建筑，但负责在现场按照蓝图调度工人、执行工序、保障安全。

这种双重身份——中间层的硬件抽象 + 高层的本地程序执行——是架构上的深思熟虑：

- **中间层身份**：Terminal 产品可能有几十种型号的传感器和操作器，每种型号的寄存器定义、通信协议、数据格式各不相同。Bastion 的 HAL 层将这些差异屏蔽，为上层提供统一的"传感器""操作器"抽象
- **高层身份**：实时控制回路（100Hz-10kHz）无法容忍 Hermes 往返延迟，Bastion 必须本地执行 Hermes 预先下发的控制程序

**关键区分**：Bastion 有执行自主性，但没有创造自主性。它可以调度任务、协调 Terminal、强制安全规则、执行降级程序——但所有这些行为的规则都由 Hermes 设计并打包下发。Bastion 不能自创控制策略、不能修改行为包参数、不能改变运营目标。

#### 执行自主性的边界

"有执行自主性但没有创造自主性"在具体场景中的界定：

**Bastion 可自主决策的场景**（执行自主性）：

| 场景 | Bastion 行为 | 规则来源 |
|---|---|---|
| 任务调度 | 按 schedules/ 调度行为包执行顺序和优先级 | 行为包中的 schedules |
| 安全规则触发 | 检测到物理量越限时立即覆盖控制输出 | 行为包中的 safety_rules |
| Terminal 通信异常 | 检测到 Terminal 无响应，标记故障并上报 | HAL 驱动中的超时配置 |
| 降级响应 | 检测到 Hermes 心跳丢失，按梯次执行降级 | 行为包中的 fallback |
| 行为包热替换 | 按流程加载新行为包、状态迁移、回滚 | 行为包管理模块（见模块设计文档） |
| 传感器数据合理性校验 | 过滤明显异常的读数（超出物理可能范围） | HAL 中的量程配置 |

**Bastion 不可自主决策的场景**（无创造自主性）：

| 场景 | 为什么不能自主 | 正确做法 |
|---|---|---|
| 修改控制律参数 | 即使环境变化也不可自行调整 PID 参数 | 上报 Hermes，等待新行为包 |
| 创造新的应急方案 | 遇到未预设的故障场景不能即兴创造响应 | 执行最保守的预设降级程序，上报 Hermes |
| 改变运营目标 | 不能自行从"巡检模式"切换到"作业模式" | 等待 Hermes 指令 |
| 绕过安全规则 | 即使看似"合理"也不能绕过安全约束 | 安全规则不可覆盖，上报 Hermes |
| 新增 Terminal 类型 | 检测到未注册的 Terminal 不能自动接入 | 拒绝并上报，等待 Hermes 配置 |
| 修改行为包内容 | 不能就地修改已加载行为包的任何内容 | 只能接受 Hermes 下发的新版本 |

**模糊地带**：传感器数据超出历史范围但未触发安全阈值——Bastion 可按行为包 schedules/ 中预设的策略处理（如触发额外采样、提高上报频率），但不可自行决定新的处理逻辑。

**四种物理形态**对应四种部署环境：

| 形态 | 代号 | 设计哲学 |
|---|---|---|
| 陆地型 | Ground | 稳定、可维护，性能优先。地面部署没有严格的体积和功耗限制，是功能最完整的形态 |
| 天空型 | Aerial | 轻量化、低功耗。每一克重量都意味着更高的能源成本，设计必须在能力与重量间取得平衡 |
| 潜水型 | Submarine | 耐压密封，通信受限。水下电磁波衰减极快，通信策略需要根本性的重新设计 |
| 极端环境型 | Extreme | 可靠性至上。高温、高压、强辐射、深空——任何一种都足以摧毁常规电子设备，设计优先考虑生存 |

四种形态共享同一套软件架构（HAL / engine / system / bridge），差异体现在物理设计（CAD/CAE/produce）和配置参数上。

#### Bastion 的程序执行引擎

Bastion 的核心是**程序执行引擎**（engine/），取代了早期的"AI 运行时"概念。引擎不创造智能，只执行 Hermes 下发的行为包：

- **模型执行**（model_exec/）：运行 Hermes 预编译的推理模型（ONNX/TFLite/TensorRT 格式）
- **信号处理**（signal_proc/）：执行 Hermes 定义的信号处理流水线
- **控制执行**（control_exec/）：执行 Hermes 设计的控制律（PID 参数、MPC 配置、状态机）
- **安全规则**（safety/）：强制执行 Hermes 定义的安全约束，可覆盖控制执行命令以保护硬件
- **行为包管理**（behavior_pack/）：加载、校验、版本管理、热替换 Hermes 下发的行为包

### Terminal 终端 — 底层

> 终端——一切链路的终点，也是一切行动的起点。

Terminal 是 Thaumazein 中**数量最多、分布最广**的产品。它是唯一与物理世界直接接触的节点，核心能力是**电磁波与物理量之间的纯粹转换**——不做推理，不做预处理，不做决策。

**传感器**（Sensor）将物理世界的信号——力、温度、磁场、光强、声波——转换为电磁信号上报给 Bastion。

**操作器**（Operator）将 Bastion 下发的电磁信号转换为物理世界的作用——力、扭矩、磁、电、光。

传感器与操作器并非严格分离。一个感知-操作一体终端可以同时监测和操控同一个物理量，形成硬接线的本地回路（传感器信号直接触发操作器输出），闭环控制逻辑由 Hermes 设计、Bastion 执行。

Terminal 的设计哲学是**极简而专一**。每个终端只做一件事：信号转换。越简单，越可靠，越便宜，越适合大规模部署。

## 通信拓扑

```
Hermes ←──双向──→ Bastion (Hub) ←──双向──→ Terminal (Node)
  高层              中间层+高层            底层(物理)
```

### Hermes ↔ Bastion 通信

| 接口 | 方向 | 内容 | 说明 |
|---|---|---|---|
| 行为包部署 | 下行 | 完整的行为包（控制律、模型、安全规则） | Bastion 校验后加载，拒绝未签名或格式错误的行为包 |
| 运行时配置 | 下行 | 参数更新、模式切换 | Bastion 确认并在超时内应用 |
| 控制指令 | 下行 | 高层命令 | Bastion 翻译为执行引擎动作 |
| 遥测流 | 上行 | 执行指标、安全事件 | 按可配置频率批量上报 |
| 经验数据 | 上行 | 聚合的传感器数据、异常报告 | 按 Hermes 调度缓冲上传 |
| 心跳 | 双向 | 健康状态 | 双方监控；链路丢失触发 Bastion 降级程序 |

### Bastion ↔ Terminal 通信

| 接口 | 方向 | 内容 | 说明 |
|---|---|---|---|
| 传感器轮询 | 下行 | 轮询命令 | 寄存器级操作，Terminal 在硬件规定时间内响应 |
| 传感器数据 | 上行 | 原始传感器读数 | ADC 值、数字读数等原始信号，不做解读 |
| 操作器命令 | 下行 | 执行器命令 | 寄存器级操作，Terminal 确认后执行 |
| 操作器状态 | 上行 | 执行器状态 | 位置、电流、状态位，简单上报不做分析 |
| 终端健康 | 上行 | 电源、连接、自检结果 | 状态位，变更时或周期性上报 |

## 多节点拓扑

### Hermes-Bastion 拓扑（一对多）

```
                ┌── Bastion A (Ground)
                │
Hermes ─────────┼── Bastion B (Aerial)
                │
                └── Bastion C (Submarine)
```

- **连接模型**：星型。每个 Bastion 独立与 Hermes 建立 TLS 隧道，Bastion 之间默认无直接通信
- **容量**：Hermes 可管理的 Bastion 数量受限于：带宽（遥测流聚合带宽）、计算（AI 推理并发）、人类监控能力（dashboard 可视化上限）。初始设计目标：单 Hermes 管理 ≤ 32 个 Bastion
- **寻址**：每个 Bastion 有全局唯一的 `bastion_id`，在 Hermes 的 CA 签发证书时绑定

### Bastion-Terminal 拓扑（一对多）

```
                    ┌── Terminal 1 (Sensor)
                    │
Bastion (Hub) ──────┼── Terminal 2 (Operator)
                    │
                    └── Terminal N (Sensor-Operator)
```

- **连接模型**：总线或点对点，取决于物理层（CAN 总线 / UART 点对点 / SPI 总线）
- **容量**：按物理形态不同：
  - Ground：≤ 128 Terminal（CAN 总线地址空间）
  - Aerial：≤ 16 Terminal（重量和功耗限制）
  - Submarine：≤ 32 Terminal（带宽限制）
  - Extreme：≤ 8 Terminal（可靠性优先，节点越少越好）
- **寻址**：每个 Terminal 在注册阶段由 Bastion 分配 `terminal_id`，仅在 Bastion 管辖范围内唯一

### Bastion 间协同

Bastion 之间默认不直接通信。需要多机协同控制时（如多 Bastion 协作搬运）：

- **协调者**：Hermes 作为唯一的协调者，向每个参与的 Bastion 分别下发协同行为包
- **同步机制**：每个协同行为包包含共享的时间基准和同步点，Bastion 在达到同步点时等待对齐
- **不直连的原因**：保持架构简洁，避免引入 Bastion 间的信任关系和安全复杂性。如果未来性能需要，可增加 Bastion 间的高速数据链路（需额外的安全设计）

### Hermes 高可用

Hermes 是架构中的单点。其不可用会导致所有 Bastion 进入降级模式，系统丧失智能更新能力。

**降级容忍度**：
- 短期中断（< 30min）：Bastion 按 fallback/ 程序继续运行，影响可控
- 中期中断（30min - 24h）：Bastion 无法获取新的行为包或控制指令，无法响应环境变化
- 长期中断（> 24h）：需要人工介入

**高可用方案（按优先级）**：

| 方案 | 保护层级 | 适用阶段 |
|---|---|---|
| Hermes 热备 | 主备自动切换，Bastion 无感知 | 生产部署 |
| Hermes 冷备 | 备机手动接管，Bastion 需重连 | 成本敏感场景 |
| 多 Hermes 实例 | 多活部署，Bastion 分组归属 | 大规模部署 |

**恢复同步**：Hermes 恢复后需与 Bastion 重新同步：

1. Bastion 上报当前状态（活跃行为包、运行时状态、累计遥测）
2. Hermes 对比行为包版本，决定是否需要更新
3. 如有更新，按正常行为包部署流程下发

## 设计-实现对应性追踪

方法论的"设计与实现一一对应"约束要求每个设计元素都有对应实现。为确保此约束可执行，每个设计文档必须包含**设计-实现映射表**：

### 设计-实现映射

| 设计元素 | 四因维度 | 对应实现 | 实现位置 | 验证方式 |
|---|---|---|---|---|
