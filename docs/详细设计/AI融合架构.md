# AI 融合架构

四阶段模型和知识流概述见 [架构设计](../架构设计.md) 的 AI 四阶段章节。本文档聚焦 AI 在每个阶段的具体能力边界、知识流细节和交互接口。

---

## 一、AI 能力边界

### 设计态（Hermes）

AI 在此阶段拥有最大自由度，核心能力：

| 能力 | 说明 | 约束 |
|---|---|---|
| 需求理解与澄清 | 从自然语言或模糊描述中提取结构化需求 | 人类确认后才进入下一环节 |
| 架构生成 | 根据需求生成候选系统架构和行为包设计 | 人类选择方案，AI 不自主定案 |
| 控制律设计 | 设计 PID/MPC/状态机等控制策略 | 必须通过仿真验证 |
| 推理模型训练 | 训练 ONNX/TFLite/TensorRT 格式的推理模型 | 必须通过安全审计 |
| 安全规则生成 | 根据物理约束生成安全阈值和紧急程序 | 必须通过 safety_auditor 模块审计 |
| 降级程序设计 | 为链路降级/丢失/长期中断设计响应策略 | 必须覆盖所有已知故障模式 |

**关键约束**：AI 的所有产出物必须经过人类审核 + 仿真验证 + 安全审计，才能进入部署态。AI 不自主决定部署。

### 部署态（Hermes → Bastion）

AI 不参与此阶段。部署是确定性的机械操作：

1. Hermes 将验证通过的产物打包为行为包（签名、校验）
2. 通过 bridge/ 传输到 Bastion
3. Bastion 校验后加载到 staging 区

### 执行态（Bastion）

Bastion 确定性执行行为包，不运行任何 AI 推理：

- 模型执行：运行预编译模型，输入→输出的确定性映射
- 控制执行：按预设参数执行控制律，参数不可被 Bastion 修改
- 信号处理：按预设流水线处理，流水线不可被 Bastion 重组

**"确定性执行"的含义**：相同输入 + 相同行为包 = 相同输出。Bastion 不引入随机性、不自行调整参数。

**精度约束**：推理模型的浮点运算在不同硬件（CPU/GPU/NPU）上可能存在微小差异。对于硬实时控制场景（control_exec/），行为包应限定使用定点数模型或 CPU-only 执行，以保证跨平台的比特级确定性。模型执行（model_exec/）允许浮点运算，但需在行为包 manifest 中标注精度等级：`deterministic`（定点/CPU，比特级一致）或 `approximate`（浮点/GPU，允许 ULP 级差异）。

### 反馈态（Bastion → Hermes）

Bastion 采集的数据作为 Hermes AI 的输入，但数据流是单向的：

```
Bastion（采集）──遥测/经验数据──→ Hermes knowledge/
                                   │
                                   ▼
                              AI 分析与归纳
                                   │
                                   ▼
                         更新知识库 → 影响下次设计态
```

**Bastion 不做**：数据解读、异常归因、经验总结。只采集和上报。

**Hermes AI 在反馈态的能力**：
- 异常模式识别（从遥测流中发现异常模式）
- 控制性能评估（对比预期行为与实际行为）
- 知识沉淀（将运行经验归纳为可复用的设计知识）
- 模型优化（基于实际数据重新训练或微调模型）

## 二、知识流

### 知识类型

| 知识类型 | 来源 | 存储位置 | 用途 |
|---|---|---|---|
| 设计知识 | 设计态 AI 产出 | knowledge/ | 供未来设计复用（架构模式、控制策略、安全规则模板） |
| 运行经验 | 反馈态数据归纳 | knowledge/ | 修正设计假设（实际功耗、实际延迟、故障频率） |
| 物理模型 | 仿真验证 | model/simulation/ | 作为仿真基础，校准数字孪生 |
| 安全基线 | safety_auditor 审计 | knowledge/ | 安全规则的有效性验证，发现盲区 |

### 知识回路

```
设计态 ──产出──→ 行为包 ──部署──→ 执行态
   ↑                                 │
   │                            采集遥测/经验
   │                                 │
   │                                 ▼
   └── 知识沉淀 ←── 归纳分析 ←── 反馈态
```

知识回路的关键特征：
- **异步性**：反馈不即时影响正在执行的行为包。知识沉淀影响的是**下一次**设计
- **版本化**：每次知识更新关联到具体的行为包版本，支持回溯
- **可审计**：AI 的知识归纳过程可追溯，人类可审查 AI 的推理链路

### 知识到设计的映射

Hermes 的 designer 模块从知识库中检索相关设计知识，辅助新行为包的设计：

1. 根据新需求检索相似的历史设计（架构模式、控制策略）
2. 根据运行经验校准设计参数（实际环境与仿真假设的偏差）
3. 根据安全基线检查安全规则覆盖度

## 三、Hermes AI 内部架构

```
agent/
├── designer/        ──→ 连接需求与实现，调用 generator 和 knowledge
├── generator/       ──→ 将设计方案转化为可执行产物（代码、模型、行为包）
├── knowledge/       ──→ 知识库管理与检索，经验数据归纳
└── safety_auditor/  ──→ 独立于 designer，审计安全规则完整性和正确性
```

**独立性约束**：safety_auditor 与 designer 是独立的模块。安全审计不由设计 AI 自己执行，避免利益冲突。这类似于代码审查不能由作者自己完成。

**可信度保障**：safety_auditor 的角色是**人类安全审查的辅助工具**，而非独立审计者。两者的底层能力相同（串行概率预测），存在对同一类推理盲区的系统性风险。因此：

1. **安全规则的类别和底线由人类主导定义**——"需要保护哪些物理量""不可逾越的红线是什么"。AI 在人类定义的框架内填充具体参数和检查覆盖度
2. **safety_auditor 检查的是结构完整性和一致性**——每个执行器是否有安全规则覆盖、规则之间是否矛盾、保护动作是否可达。这些是可形式化验证的检查项
3. **安全规则的最终审核权在人类**——AI 的安全审计结果是建议，人类审查并确认后才进入部署态。不可将安全审计的决策权完全交给 AI
