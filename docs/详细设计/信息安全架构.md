# 信息安全架构（Security）

本文档覆盖**信息安全（Security）**。功能安全（Safety）见 [功能安全设计](功能安全设计.md)。

## 功能安全 vs 信息安全

| 维度 | Safety（功能安全） | Security（信息安全） |
|---|---|---|
| 威胁来源 | 物理故障、软件缺陷、环境异常 | 恶意攻击者、未授权访问、数据篡改 |
| 防护目标 | 人员与设备不受物理伤害 | 系统不被入侵、数据不被窃取篡改 |
| 设计原则 | Hermes 定义安全规则，Bastion 强制执行 | 纵深防御，每层独立承担安全责任 |
| 文档 | [功能安全设计](功能安全设计.md) | 本文档 |

二者独立但互补：Security 失败可能导致 Safety 规则被绕过，Safety 机制不能替代 Security 防护。

## 一、威胁模型

### 信任边界

```
┌─────────────────────────── 信任域：Hermes ──────────────────────────┐
│  Hermes 是信任锚点。行为包在此创建、签名、分发。                      │
│  假设：Hermes 运行环境受控，物理访问受限。                            │
└─────────────────────────── ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┘
                              │ 加密隧道
┌─────────────────────────── 半信任域：Bastion ───────────────────────┐
│  Bastion 在边缘部署，可能暴露于物理攻击。                             │
│  假设：攻击者可物理接触 Bastion，但无法提取密钥（硬件安全模块）。      │
│  信任：Bastion 忠实执行 Hermes 签名的行为包。                        │
└─────────────────────────── ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┘
                              │ 明文/轻量加密
┌─────────────────────────── 非信任域：Terminal ──────────────────────┐
│  Terminal 大量部署，物理完全暴露。                                    │
│  假设：攻击者可物理接触、替换、篡改 Terminal。                        │
│  信任：Terminal 不被信任，Bastion 必须验证 Terminal 数据合理性。      │
└──────────────────────────────────────────────────────────────────────┘
```

### 威胁清单

| 编号 | 威胁 | 影响层 | 严重度 | 缓解措施 |
|---|---|---|---|---|
| T1 | 伪造行为包注入 Bastion | Hermes→Bastion | 高 | 行为包签名验证 |
| T2 | 中间人攻击 Hermes↔Bastion 通信 | 通信链路 | 高 | TLS 加密隧道 |
| T3 | 重放旧行为包或控制指令 | 通信链路 | 中 | 序列号 + 时间戳窗口 |
| T4 | 篡改 Terminal 固件 | Terminal | 高 | Secure boot + 固件签名 |
| T5 | 伪造 Terminal 数据欺骗 Bastion | Bastion←Terminal | 中 | 数据合理性校验 + 异常检测 |
| T6 | 物理接触提取 Bastion 密钥 | Bastion | 高 | 硬件安全模块 (HSM) |
| T7 | 拒绝服务（洪泛通信链路） | 通信链路 | 中 | 速率限制 + 优先级队列 |
| T8 | 未授权终端接入网络 | Bastion↔Terminal | 中 | 终端注册与认证 |

## 二、身份认证

### 2.1 Hermes ↔ Bastion 双向认证

- **协议**：TLS 1.3，双向证书认证（mTLS）
- **证书体系**：
  - Hermes 作为根 CA，签发 Bastion 设备证书
  - Bastion 证书包含 `bastion_id` 和 `bastion_type`（物理形态）
  - 证书有效期：建议 1 年，支持吊销（CRL/OCSP）
- **密钥存储**：
  - Hermes：私钥存储在受控环境中
  - Bastion：私钥存储在 HSM 或 TPM 中，不可导出

### 2.2 终端注册

Terminal 无力运行 TLS，采用轻量级注册机制（帧格式定义见 [通信设计](通信设计.md) §2.6）：

- **注册阶段**：Terminal 首次接入时发送 REGISTER 帧（含设备型号、序列号），Bastion 校验白名单后通过 REGISTER_ACK 帧分配 `terminal_id` 并写入共享密钥
- **运行阶段**：注册完成后，T→B 帧和 ACTUATOR_CMD 帧携带 HMAC-SHA256 截断（4 字节），覆盖 LEN + TYPE + PAYLOAD + CRC。HMAC 算法：HMAC-SHA256(shared_key, frame_without_hmac_and_tail)，取前 4 字节
- **防重放**：PAYLOAD 中包含按帧类型定义的序列号或时间戳字段，Bastion 跟踪最近序列号拒绝重复

```
Terminal                     Bastion
  │                            │
  │── 1. REGISTER ───────────→│  含 VENDOR/MODEL/SERIAL/CAPABILITIES
  │                            │  校验设备白名单
  │← 2. REGISTER_ACK ────────│  分配 terminal_id + HMAC 共享密钥
  │                            │
  │── 3. 正常帧(含 HMAC) ────→│  每帧校验 HMAC
```

- **密钥轮换**：Bastion 通过 KEY_UPDATE 帧下发新密钥（见 [通信设计](通信设计.md) §2.7）

## 三、通信加密

### 3.1 Hermes ↔ Bastion

- **传输层**：TLS 1.3 加密隧道
- **前向保密**：强制 ECDHE 密钥交换
- **密码套件**：优先 AES-256-GCM + ChaCha20-Poly1305

### 3.2 Bastion ↔ Terminal

Terminal 资源受限，不使用 TLS。安全策略按物理形态分级：

| Bastion 形态 | 威胁环境 | 加密策略 |
|---|---|---|
| Ground | 有线/短距无线，相对安全 | 帧级 HMAC 校验，数据明文 |
| Aerial | 无线通信，易被窃听 | AES-128-CCM 加密 + HMAC |
| Submarine | 水声通信，带宽极低 | HMAC 仅校验完整性，不加密（带宽优先） |
| Extreme | 高威胁环境 | AES-256-GCM 加密 + HMAC + 防重放 |

## 四、行为包安全

### 4.1 签名流程

```
Hermes (签名)                           Bastion (验签)
  │                                        │
  │  1. 构建行为包归档                      │
  │  2. 计算 SHA-256 digest                │
  │  3. 用 Hermes 私钥签名 (Ed25519)       │
  │  4. 附加 signature 文件                │
  │── 5. 传输 ────────────────────────────→│
  │                                        │  6. 用 Hermes 公钥验签
  │                                        │  7. 校验 checksum
  │                                        │  不通过 → 拒绝 + 告警
```

### 4.2 签名文件格式

```json
{
  "algorithm": "Ed25519",
  "public_key_fingerprint": "string, Hermes 签名公钥指纹",
  "signature": "string, base64 编码的签名值",
  "signed_digest": {
    "algorithm": "SHA-256",
    "excludes": ["signature"]
  }
}
```

### 4.3 密钥管理

- **Hermes 签名密钥对**：Ed25519，定期轮换（建议 90 天）
- **Bastion 预置**：出厂时写入当前 Hermes 签名公钥 + CA 证书
- **密钥更新**：通过签名的行为包通道分发新公钥（自举信任链）
- **紧急吊销**：Hermes 可下发证书吊销列表（CRL），Bastion 在心跳周期内更新

## 五、固件安全

### 5.1 Terminal Secure Boot

Terminal 资源有限，采用轻量级 secure boot：

1. **Bootloader**：只读存储（ROM 或写保护 Flash），包含 Hermes 公钥哈希
2. **固件签名**：每个固件镜像由 Hermes 私钥签名
3. **启动流程**：Bootloader 校验固件签名 → 通过则加载 → 失败则停留于 Bootloader 等待 Bastion 重刷
4. **回滚保护**：固件头包含单调递增版本号，拒绝降级

### 5.2 Bastion Secure Boot

Bastion 算力充足，采用完整 secure boot：

1. **硬件信任锚**：TPM/HSM 存储 Bastion 根密钥
2. **启动链**：Bootloader → 内核 → 应用，每级验证下级签名
3. **远程证明**：Bastion 启动后向 Hermes 报告启动度量值，Hermes 验证完整性

## 六、纵深防御策略

每层独立承担安全责任，不依赖上层防护：

| 层次 | 安全职责 | 失败时的应对 |
|---|---|---|
| Hermes | 签名行为包、签发证书、审计日志、密钥管理 | 人工介入 |
| Bastion | 验证行为包签名、TLS 终端、Terminal 认证、数据合理性校验 | 拒绝未认证指令，触发 Safety 降级 |
| Terminal | Secure boot、帧级 HMAC、序列号防重放 | 拒绝未认证命令，停留安全状态 |

**关键原则**：Security 失败不应绕过 Safety。即使攻击者成功注入伪造数据，Bastion 的 Safety 引擎仍基于 Hermes 预设的物理约束（力矩上限、速度上限）强制执行硬保护。
