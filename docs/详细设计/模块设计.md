# 模块设计

三大产品线的模块划分、目录结构与核心能力。

## 1. Hermes 产品 `product/hermes/` — 高层

面向人类的设计态工程环境，创造所有智能产物。

### 模块划分

```
product/hermes/
├── intent/                  # 意图理解
│   ├── requirement/         # 需求捕获与管理
│   └── goal/                # 目标分解与追踪
├── model/                   # 系统建模
│   ├── architecture/        # 系统架构编辑器
│   ├── interface/           # 接口定义与契约管理
│   ├── simulation/          # 仿真模型管理
│   └── behavior_design/     # 行为包设计工具
├── agent/                   # AI 设计助手
│   ├── designer/            # AI 设计助手（核心）
│   ├── generator/           # 代码/方案/行为包生成引擎
│   ├── knowledge/           # 知识库与经验管理
│   └── safety_auditor/      # 安全规则审计
└── dashboard/               # 综合控制台
    ├── monitor/             # 运行态监控（含行为包执行状态）
    ├── analytics/           # 数据分析
    └── deployment/          # 部署管理（含行为包生命周期）
```

### 核心能力

| 模块 | 职责 | AI 参与 |
|---|---|---|
| intent | 从自然语言或结构化输入捕获需求 | AI 理解意图，补全和澄清需求 |
| model | 系统建模、架构可视化、行为包设计 | AI 根据需求生成候选架构和行为包 |
| designer | 设计助手，连接需求与实现 | AI 自主设计方案，人类审核 |
| generator | 将设计转化为行为包等可执行产物 | AI 自动生成代码、配置、仿真脚本 |
| knowledge | 管理设计知识和运行经验 | AI 提取和归纳知识 |
| safety_auditor | 审计安全规则的完整性和正确性 | AI 检查安全规则漏洞和边界情况 |
| dashboard | 监控、分析、部署的统一入口 | AI 提供异常检测和优化建议 |

## 2. Bastion 产品 `product/bastion/` — 中间层 + 高层

边缘程序执行中心，担当"包工头"角色：按 Hermes 的蓝图调度 Terminal、执行程序、保障安全。

### 模块划分

```
product/bastion/
├── physics/                 # 硬件设计
│   ├── cad/                 # CAD 设计图纸
│   ├── cae/                 # CAE 仿真流程自动化脚本
│   └── produce/             # 制造流程文档与工艺参数
└── codes/                   # 软件设计
    ├── hal/                 # 中间层核心：硬件抽象层
    │   ├── sensor_driver/   # 传感器型号驱动（C/汇编）
    │   ├── actuator_driver/ # 操作器型号驱动（C/汇编）
    │   └── comm/            # 通信接口（UART/SPI/I2C/USB/网络）
    ├── engine/              # 高层能力：程序执行引擎
    │   ├── model_exec/      # 执行 Hermes 下发的推理模型
    │   ├── signal_proc/     # 执行 Hermes 下发的信号处理流水线
    │   ├── control_exec/    # 执行 Hermes 下发的控制律
    │   ├── behavior_pack/   # 行为包生命周期管理（加载/校验/版本/热替换）
    │   └── safety/          # 安全规则强制执行引擎
    ├── system/              # 系统管理
    │   ├── task/            # 任务调度
    │   ├── config/          # 配置管理
    │   ├── update/          # 远程更新
    │   └── fallback/        # 降级程序管理（Hermes 不可达场景）
    └── bridge/              # 与 Hermes 的桥接
        ├── sync/            # 数据同步
        ├── protocol/        # 通信协议
        ├── pack_deploy/     # 行为包部署接口
        └── telemetry/       # 执行遥测上报
```

### 核心能力

- **硬件抽象层 (HAL)**：以 C/汇编语言实现不同型号传感器与操作器的驱动抽象，为上层提供统一接口，屏蔽底层物理差异
- **程序执行引擎 (engine)**：加载并运行 Hermes 下发的行为包——模型执行、信号处理、控制执行、安全规则强制执行
- **行为包管理**：接收、校验、版本管理、热替换 Hermes 部署的行为包
- **降级程序**：Hermes 不可达时，按预设梯次降级（链路降级 → 链路丢失 → 长期中断）
- **桥接模块**：与 Hermes 双向通信——行为包部署、遥测上报、经验数据回传

### 内部数据流

```
┌─────────────────────────────────────────────────────────────┐
│                        bridge/                               │
│  pack_deploy ←── Hermes 行为包    telemetry ──→ Hermes 遥测  │
└───────────┬────────────────────────────────────────┬─────────┘
            │ 行为包加载                                │ 遥测/经验数据
            ▼                                          │
┌──────────────────────────────────────────────────────┤
│                   engine/behavior_pack/                │
│   加载、校验、版本管理、热替换                           │
└───────┬──────────┬──────────┬──────────┬─────────────┤
        │ models   │ pipelines│ control  │ safety
        │          │          │ _laws    │
        ▼          ▼          ▼          │
┌──────────────────────────────────┐      │
│       model_exec / signal_proc   │      │  safety/ 独立运行：
│       / control_exec             │      │  - 实时监控物理量
│                                  │      │  - 物理量越限 → 覆盖控制输出
│   ┌─────────────────────────┐    │      │  - 不受 control_exec 影响
│   │  控制输出 ←─────────────┼────┼──────┤
│   └─────────────────────────┘    │      │
└──────────────┬───────────────────┘      │
               │ 操作器命令                  │ 安全覆盖
               ▼                           ▼
┌──────────────────────────────────────────────────────┐
│                       hal/                            │
│   sensor_driver ──读取──→ Terminal 传感器              │
│   actuator_driver ──写入──→ Terminal 操作器            │
│   comm/ ── UART/SPI/I2C/CAN 物理层适配                │
│                                                      │
│   HAL 为 engine 提供：                                │
│   - read_sensor(terminal_id, reg_addr) → raw_data     │
│   - write_actuator(terminal_id, reg_addr, value)      │
│   - 数据合理性校验（量程检查、通信超时检测）             │
└──────────────────────────────────────────────────────┘
```

**safety 覆盖机制**：safety/ 模块在控制输出到达 HAL 之前进行拦截。当物理量越限或安全规则触发时，safety/ 直接覆盖 control_exec 的输出值（如将电机 PWM 置零），HAL 只看到 safety/ 的输出。这一机制保证：即使 control_exec 完全失效，safety/ 仍能保护硬件。

**行为包加载流程**：behavior_pack/ 将行为包各部分分发到对应执行模块——models/ → model_exec、pipelines/ → signal_proc、control_laws/ → control_exec、safety_rules/ → safety、schedules/ → system/task、fallback/ → system/fallback。

### 行为包（Behavior Pack）

Hermes 将验证过的智能产物打包为行为包，部署到 Bastion。行为包是自包含的、版本化的部署单元：

```
behavior_pack/
├── manifest.json              # 元数据：ID、版本、目标 Bastion 类型、时间戳
├── models/                    # 预编译推理模型（ONNX/TFLite/TensorRT）
├── pipelines/                 # 信号处理流水线定义
├── control_laws/              # 控制律定义（PID 参数、MPC 配置、状态机）
├── safety_rules/              # 安全规则定义（阈值、紧急程序）
├── schedules/                 # 执行调度（哪些程序何时运行）
└── fallback/                  # 降级程序（Hermes 不可达时的预设响应）
```

### 行为包生命周期

#### 状态机

```
                    Hermes 创建 & 签名
                           │
                           ▼
  ┌─────┐  传输  ┌──────────┐  校验通过  ┌─────────┐  激活指令  ┌────────┐
  │Created│──────→│ Received │──────────→│ Staged  │──────────→│ Active │
  └─────┘       └──────────┘           └─────────┘           └────────┘
                      │                      │                      │
                      │ 校验失败             │ 激活失败            │ 新版本激活
                      ▼                      ▼                      ▼
                   [已拒绝]             ┌──────────┐          ┌──────────┐
                                       │ Rejected │         │ Superseded│
                                       └──────────┘          └──────────┘
                                                                  │
                                                                  │ 退役指令
                                                                  ▼
                                                             ┌──────────┐
                                                             │ Retired  │
                                                             └──────────┘
```

- **Created**：Hermes 端已创建、签名、但尚未传输
- **Received**：Bastion 已接收归档，正在校验签名、checksum、依赖、兼容性
- **Staged**：校验通过，行为包已加载到 staging 区，可随时激活
- **Active**：正在被引擎加载执行
- **Superseded**：被新版本行为包替代，保留在本地作为回滚候选
- **Retired**：由 Hermes 下发退役指令移除，释放存储空间

#### 热替换过程

1. **准备阶段**：新行为包进入 Staged 状态，旧行为包仍在 Active
2. **状态快照**：Bastion 对当前 Active 行为包的运行时状态做快照（PID 积分项、滤波器状态、状态机当前状态等）
3. **切换窗口**：在下一个安全的时间窗口（如控制循环间隙）执行切换：
   - 旧 Active → Superseded（不卸载，保留在内存）
   - 新 Staged → Active
   - 将状态快照注入新行为包（若兼容）
4. **验证期**：新行为包运行若干周期（可配置），Bastion 监控关键指标
5. **确认或回滚**：
   - 确认：指标正常，卸载 Superseded 版本
   - 回滚：指标异常或安全规则触发，立即切回 Superseded 版本

**关键约束**：
- 切换窗口必须小于硬实时控制周期（避免控制中断）
- Safety 引擎在热替换期间保持运行，不受影响
- 回滚不依赖 Hermes 通信，纯本地操作

**热替换时序可行性**：

热替换的切换操作本质是指针交换（行为包内存区域已预加载到 Staged），实际切换耗时在微秒级。但不同控制频率下的约束不同：

| 控制频率 | 周期 | 热替换策略 |
|---|---|---|
| ≤ 100Hz（≥ 10ms 周期） | 充裕 | 标准热替换：快照 + 指针切换 + 状态注入，在控制循环间隙完成 |
| 100Hz–1kHz（1ms–10ms 周期） | 紧凑 | 快速热替换：省略状态注入，新行为包从默认状态启动，接受短暂的过渡期性能下降 |
| > 1kHz（< 1ms 周期） | 不可行 | 不支持热替换。必须停止控制循环 → 替换 → 重新启动。由 Hermes 决定是否允许停机替换 |

高频率控制循环的热替换需 Hermes 在设计时显式标注为"停机替换"或"快速替换"。

#### 依赖与冲突

**依赖声明**：行为包 manifest 中声明依赖的其他行为包 ID 和版本范围。Bastion 在校验阶段检查依赖是否满足。

**资源冲突**：当多个 Active 行为包试图操作同一资源（执行器、传感器、计算单元）时：

1. **排他性资源**（如单个执行器的位置控制）：同一时刻只允许一个行为包控制。后续激活的行为包需在 manifest 中声明 `replaces` 前任
2. **共享性资源**（如传感器读数）：多个行为包可同时读取。Bastion 的 HAL 层负责数据分发
3. **冲突裁定**：由 Hermes 在设计时解决（通过行为包的 `replaces` 和 `requires` 声明），Bastion 在加载时拒绝冲突的组合

**版本兼容**：行为包版本遵循 semver。Bastion 拒绝加载与当前运行环境或已有行为包不兼容的版本。

#### 降级行为包

每个行为包必须包含 fallback/ 目录。当 Hermes 链路中断时，Bastion 从当前 Active 行为包的 fallback/ 加载降级程序，而非独立的降级行为包。这确保降级行为与当前行为包版本一致。

## 3. Terminal 产品 `product/terminal/` — 底层

物理信号转换节点。Terminal 不做高层推理和决策——它不解读数据的含义、不自主调整行为、不参与闭环控制的逻辑判断。Terminal 的固件仅包含**确定性的信号处理**：通信帧的收发与校验、传感器信号的 ADC 采集与格式化、操作器信号的 DAC 输出，以及上电自检（bootstrap/ 对硬件寄存器的确定性检查）。这些操作不涉及推理或决策，是固定的输入-输出映射。

### 模块划分

```
product/terminal/
├── sensor/                  # 感知器终端
│   ├── firmware/            # 固件：信号采集与上报
│   ├── driver/              # 硬件信号调理
│   └── protocol/            # 通信协议栈
├── operator/                # 操作器终端
│   ├── firmware/            # 固件：命令接收与执行
│   ├── driver/              # 硬件驱动电路
│   └── protocol/            # 通信协议栈
└── common/                  # 终端公共模块
    ├── power/               # 电源管理
    └── bootstrap/           # 启动与自检
```

### 终端分类

| 终端类型 | 物理实体 | 能力 |
|---|---|---|
| 纯感知器 | 仅传感器 | 信号采集、电磁信号上报 |
| 纯操作器 | 仅执行器 | 接收命令、执行物理动作 |
| 感知-操作一体 | 传感器 + 执行器 | 双向信号转换，支持硬接线本地回路 |

感知-操作一体的"本地回路"是硬接线信号通路（传感器信号直接触发操作器输出），闭环控制逻辑由 Hermes 设计、Bastion 执行，Terminal 不参与软件层面的闭环。
