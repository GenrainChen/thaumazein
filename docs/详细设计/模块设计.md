# 模块设计

三大产品线的模块划分、目录结构与核心能力。

## 1. Hermes 产品 `product/hermes/` — 高层

面向人类的设计态工程环境，创造所有智能产物。

### 模块划分

```
product/hermes/
├── intent/                  # 意图理解
│   ├── requirement/         # 需求捕获与管理
│   └── goal/                # 目标分解与追踪
├── model/                   # 系统建模
│   ├── architecture/        # 系统架构编辑器
│   ├── interface/           # 接口定义与契约管理
│   ├── simulation/          # 仿真模型管理
│   └── behavior_design/     # 行为包设计工具
├── agent/                   # AI 设计助手
│   ├── designer/            # AI 设计助手（核心）
│   ├── generator/           # 代码/方案/行为包生成引擎
│   ├── knowledge/           # 知识库与经验管理
│   └── safety_auditor/      # 安全规则审计
└── dashboard/               # 综合控制台
    ├── monitor/             # 运行态监控（含行为包执行状态）
    ├── analytics/           # 数据分析
    └── deployment/          # 部署管理（含行为包生命周期）
```

### 核心能力

| 模块 | 职责 | AI 参与 |
|---|---|---|
| intent | 从自然语言或结构化输入捕获需求 | AI 理解意图，补全和澄清需求 |
| model | 系统建模、架构可视化、行为包设计 | AI 根据需求生成候选架构和行为包 |
| designer | 设计助手，连接需求与实现 | AI 自主设计方案，人类审核 |
| generator | 将设计转化为行为包等可执行产物 | AI 自动生成代码、配置、仿真脚本 |
| knowledge | 管理设计知识和运行经验 | AI 提取和归纳知识 |
| safety_auditor | 审计安全规则的完整性和正确性 | AI 检查安全规则漏洞和边界情况 |
| dashboard | 监控、分析、部署的统一入口 | AI 提供异常检测和优化建议 |

## 2. Bastion 产品 `product/bastion/` — 中间层 + 高层

边缘程序执行中心，担当"包工头"角色：按 Hermes 的蓝图调度 Terminal、执行程序、保障安全。

### 模块划分

```
product/bastion/
├── physics/                 # 硬件设计
│   ├── cad/                 # CAD 设计图纸
│   ├── cae/                 # CAE 仿真流程自动化脚本
│   └── produce/             # 制造流程文档与工艺参数
└── codes/                   # 软件设计
    ├── hal/                 # 中间层核心：硬件抽象层
    │   ├── sensor_driver/   # 传感器型号驱动（C/汇编）
    │   ├── actuator_driver/ # 操作器型号驱动（C/汇编）
    │   └── comm/            # 通信接口（UART/SPI/I2C/USB/网络）
    ├── engine/              # 高层能力：程序执行引擎
    │   ├── model_exec/      # 执行 Hermes 下发的推理模型
    │   ├── signal_proc/     # 执行 Hermes 下发的信号处理流水线
    │   ├── control_exec/    # 执行 Hermes 下发的控制律
    │   ├── behavior_pack/   # 行为包生命周期管理（加载/校验/版本/热替换）
    │   └── safety/          # 安全规则强制执行引擎
    ├── system/              # 系统管理
    │   ├── task/            # 任务调度
    │   ├── config/          # 配置管理
    │   ├── update/          # 远程更新
    │   └── fallback/        # 降级程序管理（Hermes 不可达场景）
    └── bridge/              # 与 Hermes 的桥接
        ├── sync/            # 数据同步
        ├── protocol/        # 通信协议
        ├── pack_deploy/     # 行为包部署接口
        └── telemetry/       # 执行遥测上报
```

### 核心能力

- **硬件抽象层 (HAL)**：以 C/汇编语言实现不同型号传感器与操作器的驱动抽象，为上层提供统一接口，屏蔽底层物理差异
- **程序执行引擎 (engine)**：加载并运行 Hermes 下发的行为包——模型执行、信号处理、控制执行、安全规则强制执行
- **行为包管理**：接收、校验、版本管理、热替换 Hermes 部署的行为包
- **降级程序**：Hermes 不可达时，按预设梯次降级（链路降级 → 链路丢失 → 长期中断）
- **桥接模块**：与 Hermes 双向通信——行为包部署、遥测上报、经验数据回传

### 行为包（Behavior Pack）

Hermes 将验证过的智能产物打包为行为包，部署到 Bastion。行为包是自包含的、版本化的部署单元：

```
behavior_pack/
├── manifest.json              # 元数据：ID、版本、目标 Bastion 类型、时间戳
├── models/                    # 预编译推理模型（ONNX/TFLite/TensorRT）
├── pipelines/                 # 信号处理流水线定义
├── control_laws/              # 控制律定义（PID 参数、MPC 配置、状态机）
├── safety_rules/              # 安全规则定义（阈值、紧急程序）
├── schedules/                 # 执行调度（哪些程序何时运行）
└── fallback/                  # 降级程序（Hermes 不可达时的预设响应）
```

## 3. Terminal 产品 `product/terminal/` — 底层

物理信号转换节点，纯信号转换，无智能。

### 模块划分

```
product/terminal/
├── sensor/                  # 感知器终端
│   ├── firmware/            # 固件：信号采集与上报
│   ├── driver/              # 硬件信号调理
│   └── protocol/            # 通信协议栈
├── operator/                # 操作器终端
│   ├── firmware/            # 固件：命令接收与执行
│   ├── driver/              # 硬件驱动电路
│   └── protocol/            # 通信协议栈
└── common/                  # 终端公共模块
    ├── power/               # 电源管理
    └── bootstrap/           # 启动与自检
```

### 终端分类

| 终端类型 | 物理实体 | 能力 |
|---|---|---|
| 纯感知器 | 仅传感器 | 信号采集、电磁信号上报 |
| 纯操作器 | 仅执行器 | 接收命令、执行物理动作 |
| 感知-操作一体 | 传感器 + 执行器 | 双向信号转换，支持硬接线本地回路 |

感知-操作一体的"本地回路"是硬接线信号通路（传感器信号直接触发操作器输出），闭环控制逻辑由 Hermes 设计、Bastion 执行，Terminal 不参与软件层面的闭环。
