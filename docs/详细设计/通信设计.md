# 通信设计

## 拓扑总览

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  Sensor   │────→│ Bastion  │────→│ Hermes   │
│ Terminal  │     │ (Hub)    │     │          │
├──────────┤     ├──────────┤     ├──────────┤
│ Operator  │←───│ Bastion  │←───│ Hermes   │
│ Terminal  │     │ (Hub)    │     │          │
├──────────┤     │          │     │          │
│感知-操作  │←──→│          │←──→│          │
│一体终端   │     │          │     │          │
└──────────┘     └──────────┘     └──────────┘

  底层(物理)       中间层+高层        高层
```

Bastion ↔ Terminal：寄存器级读写，HAL 抽象后为"传感器轮询""操作器命令"，不含语义。

Hermes ↔ Bastion：行为包部署、遥测流、控制指令、经验数据、心跳。

---

## 一、Hermes ↔ Bastion 通信

### 1.1 行为包部署

**传输方向**：Hermes → Bastion（下行）

**传输协议**：基于可靠有序的传输层（TCP/TLS 或等价协议），支持断点续传。

**行为包传输格式**：

行为包以归档格式（tar.gz / zip）传输，内含以下结构：

```
behavior_pack/
├── manifest.json              # 元数据
├── models/                    # 预编译推理模型
├── pipelines/                 # 信号处理流水线定义
├── control_laws/              # 控制律定义
├── safety_rules/              # 安全规则定义
├── schedules/                 # 执行调度
├── fallback/                  # 降级程序
└── signature                  # 行为包数字签名
```

**manifest.json schema**：

```json
{
  "id": "string, 全局唯一行为包标识",
  "version": "string, 语义化版本 (semver)",
  "target_bastion_type": "string, 目标 Bastion 物理形态 (ground|aerial|submarine|extreme)",
  "target_bastion_id": "string|null, 目标 Bastion 实例 ID，null 表示广播",
  "created_at": "string, ISO 8601 时间戳",
  "created_by": "string, Hermes 实例标识",
  "signature_algorithm": "string, 签名算法 (如 Ed25519)",
  "dependencies": [
    {
      "pack_id": "string, 依赖的行为包 ID",
      "version_range": "string, semver range"
    }
  ],
  "contents": {
    "models": ["string, 文件相对路径"],
    "pipelines": ["string"],
    "control_laws": ["string"],
    "safety_rules": ["string"],
    "schedules": ["string"],
    "fallback": ["string"]
  },
  "checksum": {
    "algorithm": "string, 如 SHA-256",
    "value": "string, hex digest"
  }
}
```

**部署流程**：

```
Hermes                         Bastion
  │                               │
  │── 1. 部署通知 (manifest) ────→│  校验签名、版本兼容性、依赖完整性
  │                               │  不通过 → 拒绝 + 错误码
  │← 2. 接受/拒绝 ───────────────│
  │                               │
  │── 3. 传输行为包归档 ────────→│  校验 checksum
  │                               │
  │← 4. 接收确认 ────────────────│  加载到 staging 区
  │                               │
  │── 5. 激活指令 ──────────────→│  热替换执行（见行为包生命周期文档）
  │← 6. 激活结果 ────────────────│
```

**错误码**：

| 错误码 | 含义 |
|---|---|
| `SIGNATURE_INVALID` | 签名验证失败 |
| `VERSION_INCOMPATIBLE` | 版本与当前运行环境不兼容 |
| `DEPENDENCY_MISSING` | 声明的依赖行为包未安装 |
| `DEPENDENCY_CONFLICT` | 依赖行为包版本冲突 |
| `CHECKSUM_MISMATCH` | 归档校验和不匹配 |
| `STORAGE_FULL` | 存储空间不足 |
| `TARGET_MISMATCH` | target_bastion_type 与本机形态不匹配 |

### 1.2 运行时配置

**传输方向**：Hermes → Bastion（下行）

**消息格式**：

```json
{
  "type": "config_update",
  "target": "string, 配置目标 (behavior_pack|engine|hal|system)",
  "config_id": "string, 配置项标识",
  "value": "any, 配置值",
  "timeout_ms": "number, Bastion 应用超时",
  "request_id": "string, 请求唯一 ID"
}
```

**响应**：

```json
{
  "type": "config_ack",
  "request_id": "string, 对应请求 ID",
  "result": "applied|rejected|timeout",
  "rejection_reason": "string|null, 拒绝原因"
}
```

**约束**：Bastion 在 `timeout_ms` 内必须响应。超时视为失败，Hermes 可重试或告警。

### 1.3 控制指令

**传输方向**：Hermes → Bastion（下行）

**消息格式**：

```json
{
  "type": "command",
  "command": "string, 命令类型 (pause|resume|emergency_stop|switch_mode|...)",
  "params": "object, 命令参数",
  "priority": "normal|high|critical",
  "timestamp": "string, ISO 8601"
}
```

**优先级说明**：
- `normal`：常规指令，排队执行
- `high`：优先执行，可中断 normal 指令
- `critical`：紧急停机等安全指令，立即执行，可打断一切非安全执行

### 1.4 遥测流

**传输方向**：Bastion → Hermes（上行）

遥测流是 Bastion 按可配置频率批量上报的运行状态数据。

**遥测消息格式**：

```json
{
  "type": "telemetry",
  "bastion_id": "string, Bastion 实例标识",
  "sequence": "number, 单调递增序列号",
  "timestamp": "string, ISO 8601",
  "interval_ms": "number, 本条遥测覆盖的时间窗口",
  "engine": {
    "behavior_packs": [
      {
        "pack_id": "string",
        "version": "string",
        "status": "active|staged|error",
        "uptime_s": "number"
      }
    ],
    "cycle_count": "number, 本窗口内执行循环总数",
    "max_cycle_time_us": "number, 最坏执行时间",
    "avg_cycle_time_us": "number, 平均执行时间"
  },
  "safety": {
    "events": [
      {
        "rule_id": "string",
        "severity": "warning|violation|emergency",
        "timestamp": "string",
        "detail": "string"
      }
    ],
    "overrides": "number, 安全引擎覆盖控制执行的次数"
  },
  "terminals": {
    "total": "number, 在线终端总数",
    "faulty": "number, 故障终端数",
    "comm_errors": "number, 通信错误计数"
  },
  "bridge": {
    "hermes_rtt_ms": "number|null, 最近一次心跳 RTT",
    "last_heartbeat": "string, 最近心跳时间",
    "link_status": "normal|degraded|lost"
  }
}
```

**上报策略**：
- 默认频率：1Hz（可由 Hermes 通过运行时配置调整）
- 安全事件：`severity=emergency` 时立即上报，不等批量窗口
- 序列号连续：Hermes 可检测丢包

### 1.5 经验数据

**传输方向**：Bastion → Hermes（上行）

经验数据是聚合后的传感器数据和异常报告，由 Hermes 调度上传，用于知识积累和模型优化。

**经验数据消息格式**：

```json
{
  "type": "experience",
  "bastion_id": "string",
  "batch_id": "string, 批次唯一 ID",
  "time_range": {
    "start": "string, ISO 8601",
    "end": "string, ISO 8601"
  },
  "data": {
    "sensor_aggregates": [
      {
        "terminal_id": "string",
        "metric": "string, 物理量名称",
        "statistics": {
          "min": "number",
          "max": "number",
          "mean": "number",
          "stddev": "number",
          "sample_count": "number"
        }
      }
    ],
    "anomalies": [
      {
        "terminal_id": "string",
        "type": "string, 异常类型",
        "detected_at": "string",
        "context": "object, 异常发生时的上下文快照"
      }
    ],
    "control_performance": [
      {
        "control_law_id": "string",
        "metric": "string, 如 settling_time, overshoot, steady_state_error",
        "value": "number"
      }
    ]
  }
}
```

**调度策略**：
- 默认：Bastion 缓冲数据，Hermes 按需拉取或定时上传
- 带宽受限时：仅上传异常数据，聚合数据延后
- 存储策略：Bastion 本地保留最近 N 小时原始数据（可配置），超期丢弃

### 1.6 心跳

**传输方向**：双向

**Bastion → Hermes**：

```json
{
  "type": "heartbeat",
  "bastion_id": "string",
  "timestamp": "string, ISO 8601",
  "status": "normal|degraded|fallback",
  "active_packs": ["string, 当前活跃行为包 ID 列表"],
  "uptime_s": "number"
}
```

**Hermes → Bastion**：

```json
{
  "type": "heartbeat_ack",
  "timestamp": "string",
  "hermes_id": "string"
}
```

**超时与降级**：
- 正常 RTT 的 2 倍未收到 ack → `degraded` 状态，缓冲遥测
- 可配置 N 秒（默认 30s）无心跳 → `lost` 状态，触发降级程序（见故障与安全文档）
- 可配置 M 分钟（默认 30min）链路丢失 → 触发最保守降级程序

## 二、Bastion ↔ Terminal 通信

### 2.1 通用帧格式

Bastion 与 Terminal 之间的通信基于统一的帧格式，具体物理层（UART/SPI/I2C/USB）由 HAL 的 comm/ 模块适配。

```
┌──────┬──────┬──────┬─────────────┬──────┬──────┐
│ 0xAA │ LEN  │ TYPE │   PAYLOAD   │ CRC  │ 0x55 │
│ 1B   │ 2B   │ 1B   │  LEN bytes  │ 2B   │ 1B   │
└──────┴──────┴──────┴─────────────┴──────┴──────┘

帧头: 0xAA (固定)
LEN:  Payload 长度 (小端序, uint16)
TYPE: 帧类型码 (见下表)
PAYLOAD: 帧负载
CRC:  CRC-16/MODBUS (覆盖 LEN + TYPE + PAYLOAD)
帧尾: 0x55 (固定)
```

**帧类型码**：

| TYPE | 名称 | 方向 | 说明 |
|---|---|---|---|
| 0x01 | POLL | B→T | 传感器轮询命令 |
| 0x02 | SENSOR_DATA | T→B | 传感器读数响应 |
| 0x03 | ACTUATOR_CMD | B→T | 操作器命令 |
| 0x04 | ACTUATOR_ACK | T→B | 操作器命令确认 |
| 0x05 | ACTUATOR_STATUS | T→B | 操作器状态上报 |
| 0x06 | HEALTH | T→B | 终端健康状态 |
| 0x07 | HEALTH_REQ | B→T | 健康状态请求 |
| 0x0F | NACK | T→B | 错误响应 |

### 2.2 传感器轮询 (POLL / SENSOR_DATA)

**POLL 帧 Payload**：

```
┌──────────┬──────────┐
│ REG_ADDR │ READ_LEN │
│ 1B       │ 1B       │
└──────────┴──────────┘
```

**SENSOR_DATA 帧 Payload**：

```
┌──────────┬──────────┬─────────────────┐
│ REG_ADDR │ DATA_LEN │     DATA        │
│ 1B       │ 1B       │  DATA_LEN bytes │
└──────────┴──────────┴─────────────────┘
```

**超时**：Terminal 须在硬件规定的响应时间内（默认 5ms，按 Terminal 型号配置）回复 SENSOR_DATA 或 NACK。超时视为通信失败，由 HAL 记录并触发安全规则。

### 2.3 操作器命令 (ACTUATOR_CMD / ACTUATOR_ACK)

**ACTUATOR_CMD 帧 Payload**：

```
┌──────────┬──────────┬─────────────────┐
│ REG_ADDR │ DATA_LEN │     DATA        │
│ 1B       │ 1B       │  DATA_LEN bytes │
└──────────┴──────────┴─────────────────┘
```

**ACTUATOR_ACK 帧 Payload**：

```
┌──────────┬──────┐
│ REG_ADDR │ CODE │
│ 1B       │ 1B   │
└──────────┴──────┘

CODE: 0x00 = OK, 0x01 = INVALID_REG, 0x02 = VALUE_OUT_OF_RANGE, 0xFF = ERROR
```

### 2.4 终端健康 (HEALTH / HEALTH_REQ)

**HEALTH 帧 Payload**：

```
┌──────────┬────────┬─────────┬──────────┐
│ TERM_ID  │ VOLTAGE│ TEMP    │ STATUS   │
│ 2B       │ 2B     │ 1B      │ 1B       │
└──────────┴────────┴─────────┴──────────┘

TERM_ID: Terminal 地址 (小端序)
VOLTAGE: 电源电压 (单位 0.01V, uint16)
TEMP: 芯片温度 (单位 1°C, int8, -128~127)
STATUS: 位域
  bit 0: power_ok
  bit 1: comm_ok
  bit 2: selftest_ok
  bit 3-7: reserved
```

**上报策略**：变更时立即上报 + 周期性上报（默认 10s，可配置）。

### 2.5 错误响应 (NACK)

**NACK 帧 Payload**：

```
┌──────────┬──────┐
│ REQ_TYPE │ CODE │
│ 1B       │ 1B   │
└──────────┴──────┘

CODE:
  0x01 = TIMEOUT
  0x02 = CRC_ERROR
  0x03 = UNKNOWN_TYPE
  0x04 = BUFFER_OVERFLOW
  0xFF = INTERNAL_ERROR
```

## 三、时序约束

| 通信链路 | 最大延迟 | 超时 | 重试 |
|---|---|---|---|
| Bastion → Terminal（轮询） | 硬件规定（默认 5ms） | 2×硬件响应时间 | 最多 3 次，间隔指数退避 |
| Hermes → Bastion（心跳） | - | 2×正常 RTT | 持续发送 |
| Hermes → Bastion（配置） | - | timeout_ms | Hermes 侧重试，最多 3 次 |
| Hermes → Bastion（行为包） | - | 传输超时（按包大小估算） | 支持断点续传 |
